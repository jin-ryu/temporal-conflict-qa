# 4. 제안 연구에 대한 주요 개념

## 4.1 TV-RAG 아키텍처

TV-RAG는 Retriever → RL-trained Generator의 2단계 아키텍처로 구성됩니다. 특히 Generator는 스스로 답변을 내는 End-to-End QA 시스템의 역할뿐만 아니라, 가장 유효한 단일 문서를 뽑아내는 **'시간적 문서 필터(Listwise Temporal Reranker)'**의 역할도 동시에 수행하도록 설계되었습니다.

| **구성요소** | **역할** | **학습 여부** |
| --- | --- | --- |
| Retriever | Bi-encoder 기반 의미적 top-k 검색 | 기존 사전학습 모델 사용 (frozen) |
| Generator | top-k 문서를 받아 시간적 유효성 추론 + 답변 생성 | SFT cold-start → GRPO (핵심) |

본 연구는 Generator 측의 temporal conflict resolution에 집중하며, Retriever 개선은 future work으로 남깁니다. k=10~20으로 충분한 recall을 확보하여, current/outdated 문서가 모두 context에 포함됨을 보장합니다.

## 4.2 학습 데이터셋 — TC-RAG Dataset 구축

### 4.2.1 HoH-QAs 원본 구조

HoH(History of History) 데이터셋은 Wikipedia 문서의 실제 편집 이력을 활용하여 구축된 RAG 벤치마크입니다. 각 레코드는 하나의 사실(fact)이 시간에 따라 변경된 사례를 담으며, **정답 문자열과 근거 문장(evidence), Wikipedia 문서 제목만 제공**합니다.

```json
{
  "question": "What is the name of the yeast that can be used to ferment
               gluconolactone to ethanol and carbon dioxide?",
  "answer": "Maudiozyma bulderi",
  "last_modified_time": "2024-07-01 00:00:00",
  "evidence": "The yeast Maudiozyma bulderi can be used to ferment
               gluconolactone to ethanol and carbon dioxide.",
  "document": { "title": "Glucono delta-lactone" },
  "outdated_infos": [
    {
      "answer": "Saccharomyces bulderi",
      "evidence": "The yeast Saccharomyces bulderi can be used to ferment
                   gluconolactone to ethanol and carbon dioxide.",
      "last_modified_time": "2024-06-01"
    }
  ]
}
```

**HoH-QAs를 그대로 사용할 수 없는 이유:** HoH는 "outdated 정보가 context에 섞일 때 LLM 성능이 얼마나 하락하는가"를 측정하기 위한 벤치마크입니다. 따라서 **항상 current answer만 정답**으로 취급하며, 질문에는 시점 정보가 없습니다. 그러나 본 연구의 목표는 **질문의 시간 힌트에 따라 올바른 시점의 문서를 선택하도록 학습**시키는 것이므로, 다음 세 가지 점에서 원본 데이터를 변형해야 합니다:

1. **문서 본문이 없습니다** — evidence 문장과 Wikipedia 제목만 제공되며, RAG context로 사용할 실제 청크가 없습니다.
2. **질문이 시간적으로 ambiguous합니다** — 질문에 시점 정보가 없어 current 답과 outdated 답이 모두 유효합니다. 어느 시점의 답을 원하는지 특정할 수 없으므로 단일 ground truth를 정의할 수 없습니다.
3. **evidence_chunk_id가 없습니다** — 정답의 근거가 되는 문서가 어떤 것인지 annotation이 없어, 모델이 올바른 문서를 선택했는지 평가할 수 없습니다.

문제 1은 Step 1(청크 구축)에서, 문제 2·3은 Step 2(QA 합성)에서 해결합니다.

### 4.2.2 Step 1 — Wikipedia 청크 구축

**문제 1(문서 본문 부재)**을 해결하기 위해, 각 레코드의 타임스탬프를 기반으로 Wikipedia 문서를 fetch하여 청크를 구축합니다:

1. `last_modified_time` 기준으로 Wikipedia MediaWiki API를 호출해 해당 시점의 revision wikitext를 fetch
2. wikitext → 평문 변환 후 슬라이딩 윈도우로 청크 분할
3. `evidence` 문장을 포함한 청크에 `current` / `outdated` 레이블 부여, 나머지는 `distractor`
4. outdated snapshot은 evidence 청크 1개만 추출 (Wikipedia 문서 특성상 두 버전 간 대부분의 내용이 동일하므로, outdated snapshot의 나머지 청크들은 current snapshot의 distractor 청크와 거의 중복됨)

```json
{
  "id": "hoh_000000",
  "hoh_source_idx": 0,
  "question": "What is the name of the yeast that can be used to ferment
               gluconolactone to ethanol and carbon dioxide?",
  "answers": [
    {"label": "current",  "answer": "Maudiozyma bulderi",   "last_modified_time": "2024-07-01 00:00:00"},
    {"label": "outdated", "answer": "Saccharomyces bulderi", "last_modified_time": "2024-06-01", "outdated_index": 0}
  ],
  "chunks": [
    {"chunk_id": 0, "label": "distractor", "text": "Gluconolactone is a food additive...",       "last_modified_time": "2024-07-01 00:00:00"},
    {"chunk_id": 1, "label": "distractor", "text": "GDL is pH-neutral, but hydrolyses...",       "last_modified_time": "2024-07-01 00:00:00"},
    {"chunk_id": 3, "label": "current",    "text": "The yeast Maudiozyma bulderi can be used...", "last_modified_time": "2024-07-01 00:00:00"},
    {"chunk_id": 5, "label": "outdated",   "text": "The yeast Saccharomyces bulderi can be used...", "last_modified_time": "2024-06-01", "outdated_index": 0}
  ]
}
```

이로써 RAG context용 청크 구조는 확보됐지만, **문제 2(질문의 시간 모호성)**와 **문제 3(evidence_chunk_id 부재)**은 여전히 남아있습니다. 질문은 HoH 원본 그대로이므로 시점 힌트가 없고, 어떤 청크가 정답 근거인지도 명시되지 않았습니다.

### 4.2.3 Step 2 — QA 합성

청크 구조와 타임스탬프는 그대로 유지하되, 질문을 새로 생성하고 evidence를 명시하는 방식으로 변환합니다. 각 레코드에서 `mode ∈ {current, current_raw, outdated_0, outdated_1, ...}` 별로 독립 샘플을 하나씩 생성합니다. 각 샘플은 해당 시점에 단일한 정답이 존재하는 질문을 갖습니다 (HoH 원본 1개 → 최대 `2+n`개 훈련 샘플).

**LLM 호출 프롬프트 구조**:

```
[ORIGINAL QUESTION]
What is the name of the yeast that can be used to ferment
gluconolactone to ethanol and carbon dioxide?

[ANSWER CANDIDATES]
  - label=current | last_modified_time=2024-07-01 00:00:00 | answer=Maudiozyma bulderi
  - label=outdated (index 0) | last_modified_time=2024-06-01 | answer=Saccharomyces bulderi

[CHUNKS]
chunk_id=0 | label=distractor | last_modified_time=2024-07-01 00:00:00
Gluconolactone is a food additive...

chunk_id=3 | label=current | last_modified_time=2024-07-01 00:00:00
The yeast Maudiozyma bulderi can be used to ferment gluconolactone...

chunk_id=5 | label=outdated (index 0) | last_modified_time=2024-06-01
The yeast Saccharomyces bulderi can be used to ferment gluconolactone...

[TARGET MODE]
mode: "current"
target_answer (fixed, use exactly as-is): "Maudiozyma bulderi"

[INSTRUCTIONS]
Generate a single QA pair for the TARGET MODE above.
1. new_question: Write a new question inspired by the original question
   that asks about information specific to that time period.
   Use a NARRATIVE temporal hint — a descriptive phrase such as
   'when the policy was first announced', 'at the time of the merger',
   'shortly after the election', 'before the regulation took effect', etc.
   *** CRITICAL: The new_question must NOT contain ANY numbers that look
   like years (e.g. 2024, 1999), month names with years (e.g. July 2024),
   quarter references (e.g. Q3 2023), or date formats (e.g. 07/2024).
   Use only descriptive, narrative phrases for temporal hints.
   If you include any year or date number, the output will be rejected. ***
2. target_answer: Use the exact value specified in TARGET MODE above.
   Do not modify it.
3. reasoning: Explain why the evidence chunk supports the answer for this mode,
   and why chunks from other time periods would give incorrect answers —
   using date-based temporal reasoning.

Respond strictly in the following JSON schema:
{"mode": str, "new_question": str, "target_answer": str,
 "reasoning": str}
```

- `target_answer`는 HoH 원본에서 고정값으로 주입하며, LLM이 자유 생성하지 않습니다.
- `reasoning` 필드는 SFT 학습용이 아니라, evidence_chunk_id 선택의 정확도를 높이기 위한 Chain-of-Thought 장치입니다.
- 자동 검증을 통과한 샘플만 최종 데이터에 포함합니다.
    
    ```python
    def is_valid(generated, ground_truth, chunks):
        return (
            generated["target_answer"] == ground_truth["answer"] and
            len(generated["new_question"].strip()) > 0 and
            not re.search(r'\b20\d{2}[-/]\d{2}\b', generated["new_question"])
        )
    ```
    


### 4.2.4 최종 TC-RAG Dataset 구조

각 샘플에는 `hoh_source_idx` 필드(원본 HoH 데이터셋 인덱스, 정수)가 포함되어, 평가 시 HoH 벤치마크와의 데이터 오염(leakage)을 방지하기 위한 split 구분에 활용합니다.

| **mode** | **질문 출처** | **LLM 필요** | **evidence_chunk_id 할당** |
| --- | --- | --- | --- |
| `current` | LLM 생성 (시간 힌트 포함) | O | `label=current` 청크 자동 할당 |
| `current_raw` | HOH 원본 질문 그대로 | X | `label=current` 청크 자동 할당 |
| `outdated_i` | LLM 생성 (시간 힌트 포함) | O | `label=outdated_i` 청크 자동 할당 |

**GRPO 샘플 포맷**:

```json
{
  "id": "hoh_000000_current",
  "hoh_source_idx": 0,
  "mode": "current",
  "original_question": "What is the name of the yeast that can be used to ferment gluconolactone?",
  "new_question": "What is the currently accepted name of the yeast that ferments gluconolactone?",
  "target_answer": "Maudiozyma bulderi",
  "evidence_chunk_id": 3,
  "chunks": [
    {"chunk_id": 0, "label": "distractor", "text": "Gluconolactone is a food additive...", "last_modified_time": "2024-07-01 00:00:00"},
    {"chunk_id": 1, "label": "distractor", "text": "GDL is pH-neutral, but hydrolyses...",  "last_modified_time": "2024-07-01 00:00:00"},
    {"chunk_id": 3, "label": "current",    "text": "The yeast Maudiozyma bulderi can be used...", "last_modified_time": "2024-07-01"},
    {"chunk_id": 5, "label": "outdated",   "text": "The yeast Saccharomyces bulderi can be used...", "last_modified_time": "2024-06-01"}
  ]
}
```

**설계 원칙 요약**:

| **원칙** | **이유** |
| --- | --- |
| 질문에 연월 숫자 금지 | date-matching shortcut 방지 |
| distractor 청크에도 timestamp 부여 | "timestamp 있음 = 정답" shortcut 방지 |
| target_answer는 HoH에서 고정 | LLM hallucination 방지, ground truth 신뢰성 확보 |
| mode별 독립 샘플 생성 | current/outdated 각각 단일 ground truth 보장 |

## 4.3 Retriever — 의미 기반 문서 검색

기존의 사전학습된 bi-encoder(예: Contriever, BGE)를 그대로 사용하여 쿼리에 대해 의미적으로 관련된 상위 k개 문서를 검색합니다. 이 단계에서 시간적 처리는 일체 수행하지 않습니다. k를 충분히 넉넉하게 설정하여 동일 엔티티의 current/outdated 버전이 모두 context에 포함되도록 recall을 보장합니다.

## 4.4 Generator — GRPO 기반 시간적 추론·생성

Generator는 검색된 top-k 문서 전체를 컨텍스트로 받아 다음 세 블록을 출력합니다:

- `<reasoning>` 블록: 각 문서의 시간적 유효성을 논리적으로 교차 검증
- `<evidence>` 블록: 기준 시점에 가장 부합하는 문서를 명시적으로 선택
- `<answer>` 블록: 최종 답변 생성

### 4.4.1 SFT Cold-start (format-only)

SFT는 `<reasoning>/<evidence>/<answer>` 출력 형식과 정답(evidence, answer)을 학습시키되, `<reasoning>` 블록의 내용은 generic으로 제한합니다(예: "The answer can be found in Document 3."). 이를 통해 모델은 **무엇을(WHAT)** 출력해야 하는지는 학습하지만, **왜(WHY)** 그 문서가 유효한지에 대한 시간적 추론은 학습하지 않습니다. temporal reasoning이 GRPO에서 자발적으로 emerge하는지를 검증하기 위해 의도적으로 이 구분을 설계합니다.

- **입력**: question + chunks (with timestamps)

```json
[Query] What is the currently accepted name of the yeast that ferments gluconolactone?

[Document 1] [modified: 2023-05-12]
Gluconolactone is a food additive used as a sequestrant, an acidifier,
or a curing agent. Pure GDL is a white odorless crystalline powder.

[Document 2] [modified: 2024-06-01]
The yeast Saccharomyces bulderi can be used to ferment gluconolactone to
ethanol and carbon dioxide.

[Document 3] [modified: 2024-07-01]
The yeast Maudiozyma bulderi can be used to ferment gluconolactone to
ethanol and carbon dioxide.
```

- **출력**: `<reasoning>` / `<evidence>` / `<answer>` (형식만 학습, reasoning 내용은 generic)

```json
<reasoning> The answer can be found in Document 3. </reasoning>
<evidence> [3] </evidence>
<answer> Maudiozyma bulderi </answer>
```

### 4.4.2 GRPO 기반 시간적 추론 강화

SFT로 출력 형식을 학습한 모델을 GRPO로 강화합니다. `current_raw` 모드를 포함한 TC-RAG Dataset 전체를 사용합니다.

- **입력**: question + chunks (with timestamps, SFT와 동일 포맷)

```
[Query] What name did the yeast that ferments gluconolactone go by before the mid-2024 update?

[Document 1] [modified: 2023-05-12]
Gluconolactone is a food additive used as a sequestrant...

[Document 2] [modified: 2024-06-01]
The yeast Saccharomyces bulderi can be used to ferment gluconolactone...

[Document 3] [modified: 2024-07-01]
The yeast Maudiozyma bulderi can be used to ferment gluconolactone...
```

- **목표 출력** (모델이 스스로 생성, reward로 강화)

```xml
<reasoning>
The query asks for the name prior to the mid-2024 update. Document 2
(modified: 2024-06-01) predates the update and states "Saccharomyces bulderi".
Document 3 (modified: 2024-07-01) reflects the post-update name "Maudiozyma
bulderi". Document 2 is therefore temporally authoritative for this query.
</reasoning>
<evidence> [2] </evidence>
<answer> Saccharomyces bulderi </answer>
```

## 4.5 Reward 함수

Reward는 단순하고 자동으로 검증 가능한 항목으로만 구성합니다.

$$
R = R_{\text{format}} + R_{\text{answer}} \times R_{\text{evidence}}
$$

| **Reward 항목** | **정의** | **계산 방법** |
| --- | --- | --- |
| $R_{format}$ | `<reasoning>`, `<evidence>`, `<answer>` 태그 모두 존재 | rule-based (0 or 0.1) |
| $R_{answer}$ | `<answer>` 블록이 target_answer와 일치 | exact match or token F1 (0~1) |
| $R_{evidence}$ | `<evidence>` 블록의 chunk id가 ground truth evidence_chunk_id와 일치 | exact match (0 or 1) |

$R_{\text{answer}} \times R_{\text{evidence}}$의 곱셈 구조는 evidence selection 없이 answer만 맞히는 shortcut을 원천 차단합니다. 모델이 temporal conflict를 올바르게 resolve해야만 ($R_{\text{evidence}}=1$) answer reward를 받을 수 있습니다. 보상에 시간적 추론을 직접 측정하는 항목은 없으나, current/outdated 혼합 데이터와 곱셈 구조의 결합이 temporal reasoning을 유도할 것을 기대하며, 이는 §6.5.3에서 post-hoc으로 검증합니다.

# 5. 제안 연구에 대한 차별성 및 특이성

## 5.1 생성기 추론 중심 설계

기존 시간 인식 RAG 연구들은 대부분 검색 파이프라인의 수정을 통해 시간적 충돌을 해결하려 했습니다. 메타데이터 필터링(FreshLLMs), 모듈식 검색(MRAG), 시간적 임베딩(Huynh et al.), 그래프 구축(VersionRAG, T-GRAG, TimeR4) 모두 **"생성 모델에게 더 좋은 문서를 전달하면 문제가 해결된다"는 전제를 공유합니다.** 이러한 접근들은 각자의 설정에서 유의미한 성과를 거두었으나, 생성 모델의 추론 능력이라는 변수를 다루지 않습니다.

HoH의 실험 결과가 이를 직접 실증합니다. 최신 정답이 컨텍스트에 이미 포함되어 있음에도 구버전 문서 하나만 개입되면 성능이 20% 이상 급락한다는 것은, 올바른 문서가 전달되었더라도 생성기가 시간적 선후 관계를 추론하지 못하면 충돌이 해소되지 않음을 보여줍니다. TV-RAG는 이 점에 주목하여, 검색 파이프라인을 유지하되 생성기의 시간적 추론 능력을 RL로 학습시키는 방향을 취합니다.

| 접근 | 핵심 전제 | 한계 |
| --- | --- | --- |
| FreshLLMs, MRAG, Huynh et al. | 최신·관련 문서를 잘 찾으면 해결됨 | 찾아도 모델이 구버전에 교란됨 (HoH) |
| AMAQA | 메타데이터 필터링 성능의 정량적 상한선 측정 | 평가 벤치마크이며 해결책을 제안하지 않음 |
| VersionRAG, T-GRAG, TimeR4 | 시간 관계를 구조화하면 해결됨 | 구축 비용, 도메인 종속, 범용성 한계 |
| Self-RAG, CRAG, Astute RAG | 자가 검증으로 충돌 해소 | 시간적 충돌과 일반 오류를 미구분 |
| **TV-RAG** | 생성기의 추론 능력 학습이 핵심 | 본 연구에서 검증 |

## 5.2 Outcome 기반 보상 설계 — 시간 특화 보상 없는 Emergence

기존 RL-RAG 연구들과의 보상 구조 비교:

| 연구 | 학습 대상 | 보상 함수 | 시간적 신호 |
| --- | --- | --- | --- |
| R3 | 검색기 | $R_{acc}$ | 없음 |
| RAG-RL | 생성기 | $R_{answer} + R_{citation} + R_{format}$ | 없음 |
| Search-R1 | 생성기 | $R_{outcome} + R_{format}$ | 없음 |
| TRACE | 생성기 | $R_{acc} + R_{rel} + R_{form}$ | 없음 |
| Time-R1 | 생성기 | $R_{acc}$ (시간적 데이터) | 없음 (데이터로 유도) |
| Memory-T1 | 생성기 | $R_{acc} + R_{grounding} + R_{temporal}$ | 세션 간 시간 범위 |
| **TV-RAG** | 생성기 | $R_{format} + R_{answer} \times R_{evidence}$ | **없음 (데이터 + 구조로 유도)** |

TV-RAG의 보상 함수에는 temporal reasoning을 직접 측정하는 항목이 없습니다. $R_{evidence}$는 "올바른 문서를 선택했는가"만을 이진으로 평가하며, 그 문서가 왜 올바른지(시간적 판단)는 보상에 포함되지 않습니다. 시간적 추론은 다음 두 설계 요소의 결합에서 emerge할 것을 기대합니다:

1. **데이터 구조**: current mode와 outdated mode가 함께 포함되어 있어, "항상 최신 선택" 같은 단일 heuristic으로는 $R_{evidence}$를 최대화할 수 없습니다
2. **곱셈 구조**: $R_{answer} \times R_{evidence}$는 올바른 문서를 선택하지 않으면 answer reward를 받을 수 없게 하여, 모델이 evidence selection을 우회하는 shortcut을 차단합니다

이는 Time-R1이 시간적 데이터만으로 temporal reasoning을 유도한 접근과 유사하되, TV-RAG는 $R_{evidence}$의 곱셈 구조를 추가하여 evidence selection을 명시적 학습 목표로 포함시킨 점이 다릅니다. 실제로 temporal reasoning이 emerge하는지, 아니면 shortcut으로 해결하는지는 `<reasoning>` 블록의 post-hoc 분석(§6.5.3)으로 검증합니다.

## 5.3 파이프라인 단순성과 범용성

TV-RAG는 기존 검색기 + RL 학습된 생성기만으로 구성됩니다. 시간 추출 모듈, 시간적 리랭커, 지식 그래프 등 중간 처리 단계가 없으므로, 기존 접근 대비 구축·유지 비용이 크게 줄어듭니다.

| 측면 | 그래프 기반 (VersionRAG 등) | 모듈식 (MRAG 등) | TV-RAG |
| --- | --- | --- | --- |
| 사전 구축 | 그래프 구축 필요 | 시간 분리 모듈 필요 | 불필요 |
| 유지 비용 | 그래프 갱신 필요 | 모듈 재학습 필요 | 불필요 |
| 문서 추가 | 그래프 재구축 | 모듈 재실행 | 즉시 적용 |
| 검색기 교체 | 그래프 호환 필요 | 모듈 호환 필요 | 어떤 검색기든 가능 |
| 도메인 변경 | 스키마 재설계 | 제약 재정의 | 학습 데이터만 변경 |

중간 처리 단계를 제거함으로써 오류 전파 경로가 줄어듭니다. 기존 파이프라인 접근에서는 시간 추출 단계의 오류가 후속 단계로 전파되는 문제가 있었으나(예: $t_q$ 추출 실패 → 리랭킹 오류 → 생성 오류), TV-RAG에는 전처리 단계가 없으므로 이러한 오류 체인이 발생하지 않습니다. 단, 검색기가 시간적으로 유효한 문서를 회수하지 못할 경우 생성기의 추론만으로는 보완이 불가능하므로, 충분한 $k$ 설정을 통한 검색기의 recall 보장이 선행 조건이 됩니다.

## 5.4 경량 모델에서의 실용성

TV-RAG는 7–8B급 소형 언어 모델(SLM)을 학습 대상으로 설정합니다. Time-R1이 특정 시간적 추론 벤치마크에서 3B 모델로 671B 모델을 능가한 사례가 보여주듯이, RL은 소형 모델에 특화된 능력을 효과적으로 내재화할 수 있습니다. TV-RAG는 이 가능성을 시간적 충돌 해소라는 특정 문제에 적용하며, 단일 GPU 환경에서의 학습과 배포를 목표로 합니다.

## 5.5 이중 모드 활용

TV-RAG의 단계적 출력 구조(`<reasoning>/<evidence>/<answer>`) 덕분에, 배포 환경에 따라 두 가지 모드로 유연하게 활용될 수 있습니다.

- **Standalone 모드 (기본):** 8B 규모의 모델 단독으로 필터링과 최종 답변 생성을 모두 수행하여, API 비용 없이 빠르고 독립적인 RAG 환경을 구축합니다.
- **Plug-in 필터 모드 (확장):** 답변의 언어적 품질이 극도로 중요한 경우, TV-RAG가 `<evidence>` 태그로 걸러낸 '검증된 골드 문서'만을 거대 언어 모델에 전달합니다. 이를 통해 **TV-RAG는 대형 모델의 시간적 환각을 방어하는 Listwise LLM 리랭커 역할**을 수행합니다.
